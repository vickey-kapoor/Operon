# Benchmark Domain Logic Evicted from the Engine Core

**Operon Engineering Notes — Week of 2026-04-21**

---

## Architectural Win of the Week

The most structurally significant change this week was the extraction of benchmark-specific policy rules out of `PolicyRuleEngine` and into a dedicated `BenchmarkRegistry` (`src/benchmarks/registry.py`). Before this refactor, adding a third benchmark to Operon would have required editing `policy_rules.py`, extending the `PageHint` enum, and touching `_seed_default_guardrails()` — three files with zero benchmark identity, all contaminated with Gmail/form domain knowledge.

The old `BENCHMARK_PLUGINS` dict in `policy_rules.py` was the symptom, but the root problem was that `PageHint` itself carried four Gmail-specific enum members (`GOOGLE_SIGN_IN`, `GMAIL_INBOX`, `GMAIL_COMPOSE`, `GMAIL_MESSAGE_VIEW`), and `benchmark_name_for_intent()` defaulted unknown intents to `FORM_BENCHMARK` rather than `None`. The memory store's page_hint scoring used Python identity (`is`) instead of value equality (`==`), which meant the switch from static enum members to `PageHint("value")` construction would have silently broken memory scoring — and did, until patched. The engine had no principled boundary between "things that are always true" and "things that are true for Gmail."

After the refactor, `BenchmarkPlugin` (`src/benchmarks/registry.py`) is a dataclass carrying `rules`, `memory_seeds`, `success_tokens`, `section_map`, `default_url`, and `task_type`. Benchmarks register themselves at import time against a global `BENCHMARK_REGISTRY` singleton. `PolicyRuleEngine.choose_action()` now resolves plugins via `BENCHMARK_REGISTRY.get_rules(benchmark_name)` and falls back to the now-empty local `BENCHMARK_PLUGINS = {}` dict. Gmail logic lives in `src/benchmarks/gmail_plugin.py`; form logic in `src/benchmarks/form_plugin.py`. The `PageHint` enum retains only truly generic members (`FORM_PAGE`, `FORM_SUCCESS`, `UNKNOWN`), and all other page states are expressed as `PageHint("value")` dynamic construction — already handled by the existing `_missing_` override. Adding a WebArena task now requires zero changes to any engine file.

## Cold Post-Mortem

**Double-increment on TYPE recovery** — `src/agent/recovery.py` was pre-incrementing the retry counter before calling `_ladder_decision`, which then incremented it again. TYPE failures consumed two retry budget units per failure. Root cause: the recovery ladder refactor added a new code path without auditing existing callers. Severity: **degraded** (TYPE-heavy tasks would hard-stop prematurely; fixed in `8eceb3c`).

**Concurrent run hint cross-contamination** — `_advisory_hints` and `_cached_decision` in `src/agent/combined.py` (the combined JSON backend) were class-level state, shared across all runs in the same process. A concurrent run could overwrite another run's hints mid-step. Severity: **blocking for concurrent runs** (single-run use would not trigger this; fixed in `8eceb3c` with per-`run_id` scoping).

**`_meaningful_progress` loop-detection disable** — `src/agent/loop.py`'s `_meaningful_progress()` was returning `is_novel_action` when screenshots were unavailable, effectively disabling the period-2 and period-3 loop detectors in headless/capture-fail scenarios. Severity: **degraded** (agent could spin on NAVIGATE actions without screenshots; fixed in `8eceb3c`).

**Path traversal in run_id handling** — `src/api/routes.py` and `src/store/run_store.py` accepted `run_id` from user input and used it directly in `os.path.join` without validation. A crafted `run_id` like `../../etc/passwd` would traverse to arbitrary filesystem paths. Severity: **blocking (security)** — fixed in `8eceb3c` with regex validation + `resolve()`/`is_relative_to()`.

**`legacy_adapter` actor_output placeholder** — `runtime/legacy_adapter.py` was constructing `actor_output` with a hardcoded `VerificationStatus.SUCCESS` placeholder instead of the real verification outcome. The `UnifiedOrchestrator` critic was therefore classifying all steps as successful at the contract layer regardless of actual outcome. Severity: **degraded** (adaptation retry logic in UnifiedOrchestrator never fired correctly; fixed in `8eceb3c`).

**Dual plugin lookup path** — `PolicyRuleEngine.choose_action()` now checks `BENCHMARK_REGISTRY.get_rules(benchmark_name)` and falls back to `self._plugins.get(benchmark_name, [])`. The fallback path exists for legacy test mocks, but it means there are two authoritative sources of plugin truth. No bug yet, but the next benchmark registration could silently end up in the wrong registry. Severity: **cosmetic / design debt**.

**HITL debounce threshold not calibrated** — `_HITL_DEBOUNCE_THRESHOLD = 2` was chosen without empirical data on false-positive rate from perception misclassification. If Gemini misclassifies a page hint as `login` on two consecutive frames, the agent pauses unnecessarily. Conversely, if a real login page appears and then is obscured, the counter resets. Severity: **cosmetic** (threshold is a module constant, easy to tune; no production data yet).

**Headed integration tests not runnable in CI** — `tests/test_upload_file_native_integration.py` requires a real Windows headed session and `OPERON_RUN_HEADED_INTEGRATION=true`. The OS picker macro code path has zero CI coverage. Severity: **cosmetic** (covered by manual testing; real regression risk exists for `os_picker_macro.py` changes).

## What's Next

The dual plugin lookup (`BENCHMARK_REGISTRY` + `BENCHMARK_PLUGINS` fallback) is the most important open seam to close: the `BENCHMARK_PLUGINS` fallback dict in `policy_rules.py` should be removed and all remaining test mocks that use it migrated to `BENCHMARK_REGISTRY` registrations, after which the engine has a single authoritative source for benchmark identity. That also unblocks adding Mind2Web and WebArena as first-class `BenchmarkPlugin` registrations and validating the end-to-end benchmark suite runner against the new datasets.
